# 2026-07-04 — Tokenless git dependency (owner-directed) + stale-branch resolution

## Summary

Owner made this repo **public** and directed every consumer (Socratic.Trade,
Congress.Trade) to switch from the private GitHub Packages registry (npm auth
via `NODE_AUTH_TOKEN`/PAT) to a **tokenless git dependency** — npm installing
directly from this repo's tarball at a pinned ref, preferring a semver-stable
tag over a moving branch.

This repo needed **no code change** to support that: `package.json` already
carries `"prepare": "npm run build"`, so `npm install
github:jaywedgeworth22/congress-trading-shared#main` already builds `dist/`
correctly on install. Proven with a clean scratch-dir install with
`NODE_AUTH_TOKEN` unset:

```bash
rm -rf node_modules package-lock.json package.json && npm init -y
npm install github:jaywedgeworth22/congress-trading-shared#main
node -e "console.log(Object.keys(require('@jaywedgeworth22/congress-trading-shared')).slice(0,5))"
```

Result: install succeeded tokenless, `dist/` was generated by the `prepare`
script, and the package's exports were importable. `npm ci` /
`typecheck` / `build` / `audit` / `pack:dry` also all pass clean on `main` as
of this commit.

## What changed here

- `README.md` — replaced the "Publish Policy: GitHub Packages" section with a
  tokenless-git-dependency section: how to pin (`#semver:^1.2.x` or an exact
  tag), how to verify a pin locally, how to cut a tag. Kept a note about the
  retired private-registry path so it isn't silently forgotten.
- `AGENTS.md` — same policy flip, framed as an owner-directed rule so future
  agents don't reintroduce `publishConfig.registry` / scoped `.npmrc` auth.
- Tag `v1.2.0` pushed on this commit (matches the already-current
  `package.json` version) so consumers have a stable semver ref to pin
  against instead of a moving `main` branch.

## What did NOT change (deliberately out of scope)

- `package.json`'s `publishConfig.registry` and `.github/workflows/publish.yml`
  are left in place. They are now dormant (nothing forces `npm publish` to run
  against them) but removing the GitHub Packages publish capability outright is
  a separate decision from "consumers should install via git" — not requested
  here. Revisit only if the owner explicitly asks to fully decommission the
  registry path.
- No `.npmrc` exists in this repo (never did) — nothing to strip here. The
  scoped-registry `.npmrc` lines being removed are in the **consumer** repos
  (Socratic.Trade, Congress.Trade), not this one.

## Stale-branch verdict (resolves backlog item t167)

Two branches predate this rollout and were evaluated per the task's explicit
instruction to check them before doing new work:

- `codex/package-git-prepare-20260629` (single commit "fix: build package for
  git installs", 2026-06-29 19:53)
- `codex/package-prepare-and-ci` (single commit "fix(package): make shared
  contract package installable", 2026-06-29 19:51)

Both diverged from `main` on 2026-06-29, **before** `usageTelemetry.ts`, the
vitest suite, `effort-issues-sync.yml`, and multiple other commits landed on
`main`. Diffing each against current `main` shows both branches would *delete*
`usageTelemetry.ts`, its 509-line test file, `vitest.config.ts`, and several
CI workflows that main has since gained — i.e. they are not "add a prepare
script on top of current work," they are frozen snapshots from before that
work existed.

More importantly, **the problem they were solving is already solved on
`main`**: `main`'s `package.json` already has `"prepare": "npm run build"`
(confirmed working above), so there is nothing left for either branch to
contribute. Rebasing either onto `main` would be pure busywork with merge risk
and no functional gain.

**Verdict: both branches are superseded/dead.** Recommend deleting them
(`git push origin --delete codex/package-git-prepare-20260629
codex/package-prepare-and-ci`) the next time someone is touching branch
hygiene in this repo; not deleted as part of this change since branch deletion
wasn't explicitly requested and this note is sufficient paper trail per
AGENTS.md's "say so explicitly" convention. This closes backlog item t167.
